# chunking_type

文本切片（Chunking）策略示例：同一批文本可用多种方式切块，便于对比效果，或接入 RAG / 向量化检索流水线。

## 目录说明

- `1-固定长度切片.py`：目标长度 + 重叠，在句末/分号附近微调截断
- `2-句子边界切片.py`：先按句拆再合并，尽量不切在句中
- `3-LLM语义切片.py`：调用大模型按语义切块（需 API）
- `4-层次切片.py`：面向 Markdown 等标题层级，小节内再控长
- `5-滑动窗口切片.py`：固定窗口 + 步长，可控制重叠
- `6-自适应切片.py`：按空行段落合并，长度带容差
- `切片策略对比测试.py`：统一加载 1～6 号模块并打印对比
- `requirements.txt`：项目依赖说明

## 环境准备

建议使用 Python 3.9 及以上版本。

```bash
python -m pip install --upgrade pip
python -m pip install -r chunking_type/requirements.txt
```

如果使用 `3-LLM语义切片.py`，或在 `切片策略对比测试.py` 中启用 LLM 语义切片分支，需要配置环境变量（与脚本中的 DashScope OpenAI 兼容接口一致）：

```powershell
$env:DASHSCOPE_API_KEY="your_key"
```

## 运行方式

在仓库根目录执行：

```bash
python chunking_type/1-固定长度切片.py
python chunking_type/2-句子边界切片.py
python chunking_type/3-LLM语义切片.py
python chunking_type/4-层次切片.py
python chunking_type/5-滑动窗口切片.py
python chunking_type/6-自适应切片.py
python chunking_type/切片策略对比测试.py
```

单独运行各脚本会执行内置示例文本；对比脚本会依次调用六种策略并输出统计信息与样例切片结果。

## 策略选型：什么场景适合用哪种切片

以下按**数据形态、目标与约束**选择策略。实际工程里，往往不是只用一种，而是组合使用（例如“先层次，再句界；先句界，再固定长”）。

### 1. 固定长度切片（`1-固定长度切片.py`）

**适合：**

- 长文、格式杂乱、没有稳定标题或段落标记；
- 需要**稳定块长**，方便控制 embedding 成本与上下文窗口；
- 希望实现简单、速度快、行为稳定可复现。

**不太适合：**

- 强依赖章节结构的法律、制度、技术手册；
- 标点稀少或长句很多的文本，仍可能出现硬切。

### 2. 句子边界切片（`2-句子边界切片.py`）

**适合：**

- 中文叙述性文本、客服记录、会议纪要、转写稿；
- 希望块内尽量由**完整句子**组成，减少“半句话”被切断；
- 原文没有明显标题结构，但标点相对规范。

**不太适合：**

- 表格、代码、清单、配置文件等非自然语言内容；
- 英文缩写、小数点、省略号很多的内容，需额外优化断句规则。

### 3. LLM 语义切片（`3-LLM语义切片.py`）

**适合：**

- 结构复杂、主题混杂、规则难以手写；
- 希望切块更接近人工理解的“语义段”；
- 可接受 API 成本、网络依赖与额外延迟。

**不太适合：**

- 大规模离线建库且成本敏感；
- 无法稳定访问模型 API 的场景。

### 4. 层次切片（`4-层次切片.py`）

**适合：**

- Markdown、知识库、技术文档、Wiki、大纲式内容；
- 文档中有明确的 `#` / `##` / `###` 层级；
- 希望切片保留“小节标题 + 正文”的结构上下文。

**不太适合：**

- 没有标题的纯连续正文；
- OCR 转写后结构丢失严重的扫描件。

### 5. 滑动窗口切片（`5-滑动窗口切片.py`）

**适合：**

- 担心关键词刚好落在块边界附近，想通过**重叠窗口**提高召回率；
- 对严格语义分段要求不高，更关注覆盖率；
- 需要一个简单、稳定、好调参的基线策略。

**不太适合：**

- 不希望重复内容过多的场景；
- 训练数据构建等对冗余较敏感的任务。

### 6. 自适应切片（`6-自适应切片.py`）

**适合：**

- 文本已经按空行分段，例如新闻、博客、文章摘录；
- 段落长短不一，希望块长围绕目标值自然浮动；
- 希望尽量保留原始段落完整性，而不是机械切分。

**不太适合：**

- 单段特别长、几乎没有空行的文本；
- 原文换行风格混乱，段落边界不稳定的内容。

## 常见组合建议

### 1. RAG 通用默认

优先考虑：

- 固定长度 + overlap
- 句子边界 + 长度上限

这两类方案通常最稳，适合做项目初版 baseline。

### 2. 文档型知识库

优先考虑：

- 层次切片优先
- 小节过长时，再做句子边界或固定长度二次切分

适合制度文档、帮助中心、产品说明书、Markdown 知识库。

### 3. 杂乱文本 / OCR / 非标准资料

优先考虑：

- 固定长度切片作为保底策略
- 抽样使用 LLM 语义切片观察效果，再决定是否上线

适合你还不确定数据质量、格式变化很大、规则难统一的场景。

### 4. 高召回优先

优先考虑：

- 滑动窗口切片
- 固定长度 + 较大 overlap

适合搜索、召回、候选集扩展等更关注“不漏”的场景。

## 项目说明

本项目更偏教学与策略对比示例，适合：

- 学习不同 chunking 方法的实现思路；
- 快速测试不同文本在不同切片策略下的效果差异；
- 作为后续 RAG / 向量检索系统中的切片模块原型。

如果你要将其接入真实业务，建议在以下几个维度继续增强：

- token 级长度控制；
- 更稳健的中英文断句；
- 表格、代码、列表等结构化内容的专门切法；
- 与 embedding 模型的召回效果联动评估。
